# 系统设计文档（技术架构与功能设计）

## 1. 总体架构（文字拓扑）

```
[Web 前端 / Postman]
        │  HTTPS JSON / SSE
        ▼
[FastAPI 单体服务]
   ├─ 用户与鉴权（MySQL: user + JWT）
   ├─ 角色元数据（MySQL: character）
   ├─ 会话与消息（MySQL: conversation, chat_message）
   ├─ 知识库元数据（MySQL: knowledge_document） + 本地文件 uploads/
   ├─ 短期记忆（Redis: chat:session:{user}:{role} 与 chat:active_roles:{user}）
   ├─ RAG 引擎（PDFIngestService：Milvus 向量 + BM25 混合检索）
   ├─ 知识图谱（KnowledgeGraphService + LightRAGService，可选）
   └─ 大模型网关（LLMService：OpenAI 兼容 HTTP 流式 / 非流式）
```

后续可演进到 API Gateway + 独立对话服务 + 独立向量索引流水线，无需更改领域模型命名。

## 2. 技术栈

| 层级 | 技术 |
| --- | --- |
| Web 框架 | FastAPI |
| 前端 | React + Vite |
| 关系库 | MySQL（SQLAlchemy ORM） |
| 缓存 | Redis |
| 向量库 | Milvus（`pymilvus`，本地 Ubuntu Docker + 阿里云远程双部署） |
| 嵌入模型 | BAAI/bge-large-zh-v1.5（dim=1024，SiliconFlow API 远程调用） |
| 重排 | BGE-rerank（预留接口与配置位） |
| 大模型 | DeepSeek-V3（SiliconFlow 在线 API）/ 本地 vLLM / SGLang |
| 评测 | RAGAS（离线脚本与说明文档） |
| 压测 | Python + `httpx` + `asyncio`（QPS 基准测试脚本） |

## 3. 核心模块

### 3.1 用户与鉴权

- `UserRepository`：注册、查询、偏好更新。
- `security`：`pbkdf2_sha256` 密码哈希；`python-jose` 签发 JWT。

### 3.2 对话编排（ChatService）

顺序：`ensure_concurrent_role_slot` → 异步检索（`ThreadPoolExecutor`）+ 等待动画 → `PDFIngestService.search_hybrid` → 读取 Redis 记忆 → `LLMService.generate` → MySQL 落库 → Redis 追加一轮。

**流式接口（默认）**：
- `POST /api/chat` → SSE 流式，逐字返回 AI 回复（打字机效果）
- `POST /api/chat/stream` → 兼容别名
- 检索阶段通过 `ThreadPoolExecutor` 并发执行，同时向前端 SSE 推送 `"正在进行检索，请稍等……"` 逐字动画
- 检索完成后发送 `replace: ""` 事件清空等待文本，再输出正式回答

**非流式接口（后台任务）**：
- `POST /api/chat/background` → 一次性返回完整 JSON，供批量处理、定时任务等后台场景使用

### 3.3 记忆服务（MemoryService）

- List：`chat:session:{user_id}:{character_id}`，`LTRIM` 保留最近 `2 * max_rounds` 条字符串。
- 并发角色控制：`chat:active_roles:{user_id}` Hash，字段为 `character_id`，值为最近活跃时间戳。

### 3.4 知识库服务（KnowledgeService）

校验扩展名 → 写入 `uploads/{character_id}/{uuid}.ext` → 插入 `knowledge_document(status=pending)`。向量化与 Milvus upsert 建议在后台任务（Celery/RQ）实现。

### 3.5 RAG 引擎（PDFIngestService）

- **Collection 命名**：`character_knowledge_{character_id}`，每个角色独立 Collection
- **向量维度**：1024（BAAI/bge-large-zh-v1.5）
- **索引度量**：COSINE（创建与搜索统一）
- **混合检索流程**：
  1. `search_vector`：Milvus ANN 向量检索（COSINE）
  2. `search_keyword`：内存 BM25 全文检索
  3. `search_hybrid`：RRF 融合排序（Reciprocal Rank Fusion），默认 Top-K = 8
- **Embedding**：通过 SiliconFlow API 远程调用，带 LRU 缓存（512 条），失败时回退 SHA256 伪向量
- **索引防重复**：首次创建 Collection 时建索引，已存在则跳过

### 3.6 LLMService

- **模型**：DeepSeek-V3（SiliconFlow 在线 API），支持 OpenAI 兼容格式
- **模式**：
  - `generate()`：非流式，一次性返回完整文本
  - `generate_stream()`：流式，Generator 逐块 yield（SSE 推送）
- **Query Rewrite**：首轮对话可选指代消解，通过 LLM 重写用户问题
- **超时**：120s（`httpx.Client`），流式逐行解析 JSON

### 3.7 知识图谱与 LightRAG（可选增强）

- **KnowledgeGraphService**：基于 `nx.MultiDiGraph` 构建实体-关系图，支持 `graph_context()` 检索相关子图
- **LightRAGService**：轻量级 RAG 增强，支持 dual-mode 检索（local + global）
- 两者均为可选模块，RAG 主链路不依赖；开启后可丰富上下文，提升复杂问题回答质量

## 4. 数据库设计（逻辑）

- `user`：账号凭证、昵称、`preferred_character_ids`（JSON 文本）。
- `character`：角色定义、提示词模板、`knowledge_base_id`。
- `conversation`：`(user_id, character_id)` 唯一会话（应用层 get_or_create），`updated_at` 在消息写入时刷新。
- `chat_message`：一问一答，含 `rag_used` 标志和 `sources` JSON。
- `knowledge_document`：上传文件元数据与索引状态。

## 5. 配置项（环境变量前缀 `RAG_`）

详见仓库根目录 `.env.example`。关键项：

| 配置项 | 说明 |
|--------|------|
| `MYSQL_DSN` | MySQL 连接串 |
| `REDIS_URL` | Redis 连接串 |
| `MILVUS_URL` | Milvus 地址（如 `http://localhost:19530`） |
| `MILVUS_DIM` | 向量维度，**1024**（匹配 bge-large-zh-v1.5） |
| `MILVUS_METRIC` | **COSINE**（创建与搜索统一） |
| `LLM_PROVIDER` | 如 `siliconflow` |
| `OPENAI_API_BASE` / `OPENAI_API_KEY` | LLM API 地址与密钥 |
| `EMBEDDING_MODEL_NAME` | 如 `BAAI/bge-large-zh-v1.5` |
| `RETRIEVAL_MODE` | `hybrid`（向量+BM25 融合） |
| `SHORT_MEMORY_ROUNDS` | 短期记忆保留轮数 |
| `MAX_CONCURRENT_ROLES_PER_USER` | 每用户最大并发角色数 |
| `UPLOAD_DIR` | 文件上传本地目录 |

## 6. 性能基准（QPS 压测）

基于 `tests/qps_test.py`（完整聊天）与 `tests/qps_retrieval.py`（纯检索）的压测结果：

| 指标 | 完整聊天（`/background`） | 纯检索 | 倍数差距 |
|------|------------------------|--------|---------|
| **QPS** | **0.86** | **12.36** | 14.4x |
| **平均延迟** | 9.51s | 0.081s | 117x |
| **成功率** | 84.0% | 100.0% | — |
| **失败原因** | LLM API 并发限流/超时 | 无 | — |

**结论**：检索层性能优秀（QPS > 10），LLM 生成是系统绝对瓶颈（占 80%+ 耗时）。

## 7. 风险与后续工作

- **LLM 瓶颈**：当前 QPS 仅 ~0.86，受 SiliconFlow API 并发限制。短期建议加 Redis 缓存；中期建议部署本地 vLLM/SGLang GPU 推理服务
- **Schema 演进**：已存在数据库需手工 `ALTER TABLE` 增加新列（`preferred_character_ids`、`updated_at`、`knowledge_document` 表等）
- **异步索引**：上传后应触发队列任务解析 PDF、切分、调用嵌入服务写入 Milvus，并回写 `status=indexed/failed`
- **连接池复用**：LLM API 当前每次新建 `httpx.Client`，建议复用连接池减少 TCP 握手开销
