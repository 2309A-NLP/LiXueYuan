# RAG 系统 QPS 压测报告

## 测试环境

- 压测这块我主要测了两个场景，用的都是 Python + `httpx` 配合 `asyncio` 做并发
- **Milvus**: localhost:19530（本地 Ubuntu Docker）
- **LLM API**: SiliconFlow（DeepSeek-V3，远程）
- **Embedding API**: SiliconFlow（bge-large-zh-v1.5，远程）
- **数据**: character_knowledge_6，561 条向量记录
- **测试角色**: 招股说明书（character_id=6）
- **压测时间**: 2026-05-06（第二次复测）

## 对比总览

| 指标 | 完整聊天 | 纯检索 | 倍数差距 |
|------|--------------------------------------|----------------|---------|
| **QPS** | **0.86** | **12.36** | **14.4x** |
| **平均延迟** | 9.51s | 0.081s | 117x |
| **中位延迟** | 8.49s | 0.036s | 236x |
| **P95 延迟** | 13.97s | 0.582s | 24x |
| **P99 延迟** | — | 0.700s | — |
| **最大延迟** | 17.22s | 3.703s | 4.6x |
| **成功率** | 84.0% | 100.0% | — |
| **失败原因** | LLM API 500 / 超时（8 次） | 无 | — |

**第一个是完整聊天的 QPS。**
我测的是非流式接口，等后端把检索和 LLM 生成全部跑完、返回完整 JSON 才算结束。控制并发用的是 `asyncio.Semaphore`，先跑了 50 个请求、同时并发 10 个，上了 JWT 鉴权。这个主要看**从发请求到拿到最终结果的总耗时**，也就是检索 + 大模型生成的整链延迟。

**第二个是纯检索的 QPS。**
这个就没走 HTTP 和 LLM 了，我直接调了 `PDFIngestService.search_hybrid()`，测的是向量检索 + BM25 + 混合排序这一块的性能。还是 `asyncio` 并发压，跑了 200 个请求、并发开到 50，看的就只是**检索链路本身的吞吐**。”

## 分阶段耗时拆解（典型请求）

| 阶段 | 耗时 | 占比 |
|------|------|------|
| Query Rewrite（指代消解） | ~1.5s | ~15% |
| Embedding（向量化） | ~0.5s | ~5% |
| Vector Search（Milvus 检索） | ~0.1s | ~1% |
| Keyword Search（BM25） | ~0.05s | ~0.5% |
| Hybrid Merge（混合排序） | ~0.02s | ~0.2% |
| LLM Generation（大模型生成） | **~8-12s** | **~80%** |
| **总计** | **~10s** | **100%** |

> 结论：**LLM 生成占 80% 以上耗时**，是系统绝对瓶颈。

---

## 核心结论

### 1. 检索层性能优秀
- 纯检索 QPS 达 **12.36**，平均延迟仅 **81ms**
- 较上次压测 QPS 提升 **20.7%**（10.24 → 12.36），延迟降低 **17%**
- Milvus 本地部署响应极快，BM25+向量混合检索 100% 成功
- 检索不是瓶颈，无需进一步优化

### 2. LLM 是系统绝对瓶颈
- 完整聊天 QPS 仅 **0.86**，被 LLM 生成时间锁死
- 8 个 500 错误均来自 **SiliconFlow API 并发限流/超时**
- 每增加 1 个并发，延迟线性增长（API 排队）
- **流式接口**（前端默认 `/api/chat`）用户体验更好，但服务端并发吞吐与非流式相近

### 3. 优化建议（按优先级排序）

| 优先级 | 方案 | 预期 QPS 提升 | 实施难度 |
|--------|------|-------------|---------|
| **高** | **部署本地 vLLM/SGLang**（GPU 推理） | **5-10x** → 5-10 QPS | 中（需 GPU） |
| **高** | **缓存常见回答**（Redis） | **10x+** | 低 |
| **已完成** | **流式接口设为默认**（`/api/chat`） | 用户体验提升，QPS 不变 | 低 |
| **中** | **LLM API 连接池复用** | 减少 TCP 握手，~10% | 低 |
| **低** | **降低 retrieval_top_k**（8→5） | 微减 token 数，~5% | 极低 |

---

## 快速验证：缓存效果预估

假设 30% 的问题重复出现，启用 Redis 缓存后：
- **命中缓存的请求**：延迟降至 ~100ms（纯检索），QPS 接近检索层上限
- **未命中缓存的请求**：仍走 LLM，~10s
- **等效 QPS**: `0.91 / 0.7 + 10.24 * 0.3 ≈ 1.3 + 3.1 = 4.4`（**4.8x 提升**）

## 接口路由调整（已完成）

为优化用户体验，已将流式接口设为默认，非流式仅用于后台任务：

| 路由 | 类型 | 用途 |
|------|------|------|
| `POST /api/chat` | **SSE 流式（默认）** | 前端正常聊天 |
| `POST /api/chat/stream` | SSE 流式（兼容别名） | 向后兼容 |
| `POST /api/chat/background` | JSON 非流式 | 后台批量处理、定时任务 |

> 前端 `sendMessage`、`sendCompare` 已统一调用 `/api/chat`（根路径流式）。
