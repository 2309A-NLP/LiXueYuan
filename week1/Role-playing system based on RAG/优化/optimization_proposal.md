## 二、问题诊断

### 2.1 上下文精度仅 42%（核心瓶颈）

**根因**：检索返回的候选片段数量过多，混入大量低相关度内容。

- 当前 `RETRIEVAL_TOP_K` 可能设置过大（推测 ≥ 16），导致召回广但噪音高
- 混合检索中关键词匹配（BM25）权重偏高（0.4），容易召回字面相关但语义无关的片段
- `eval_context_precision` 评估的是前 5 个片段，但 rerank 输入来自 16 个候选，前 5 中仍混有噪音

**证据**：第 5 题"主营业务"RAG 精度仅 20%，说明前 5 片段中只有 1 个真正相关。

### 2.2 上下文召回 72%-80%（接近达标）

**根因**：关键信息有时被排在第 6 位以后，未进入前 5 评估范围；或 chunk 切分导致答案被截断。

**证据**：第 4 题"募资用途"RAG 召回仅 0%，说明检索完全遗漏了答案所在片段。

### 2.3 响应时间 14.8s（严重超标）

**根因**：
1. LLM 生成耗时过长（DeepSeek-V3 首 token 延迟高）
2. 三路检索（向量+关键词+图谱）串行或部分串行
3. rerank 输入 16 个片段，精排计算量大
4. 无 embedding 缓存，重复问题重复编码

## 三、优化措施与实施路径

### 阶段 1：提升精度（预期：42% → 75%-85%）

| 措施 | 配置项 | 当前值 | 目标值 | 原理 |
|------|--------|--------|--------|------|
| 减少候选集 | `RETRIEVAL_TOP_K` | 16 (推测) | **8** | 减少 rerank 输入噪音 |
| 提高语义权重 | `HYBRID_VECTOR_WEIGHT` | 0.6 | **0.75** | 语义匹配更精准，降低关键词误召回 |
| 收紧精排输出 | `RERANK_TOP_K` | 5 (推测) | **3** | 只保留高置信度片段，提高精度分母质量 |
| 过滤短片段 | `MIN_CHUNK_LENGTH` | 无 | **50 字** | 过滤无意义碎片 |

**预期效果**：
- 候选集从 16→8，rerank 前 3 个的置信度提升
- 语义权重 0.6→0.75，减少 BM25 字面匹配噪音
- 精度从 42% 提升至 **75%-85%**

**风险**：召回可能轻微下降，需在阶段 2 补偿。

### 阶段 2：提升召回（预期：72%-80% → 90%-95%）

| 措施 | 配置项 | 当前值 | 目标值 | 原理 |
|------|--------|--------|--------|------|
| 增大 chunk 重叠 | `CHUNK_OVERLAP` | 0 (推测) | **100 字** | 避免答案被切分到两个 chunk 边界 |
| 放宽图谱召回 | `GRAPH_MAX_DEPTH` | 1 (推测) | **2** | Graph+RAG 多跳遍历，捕获间接关系 |
| 关键词同义词扩展 | 新增同义词词典 | 无 | 行业词典 | "营收"="营业收入"="收入"，提升关键词覆盖 |

**预期效果**：
- chunk 重叠增加，减少边界丢失
- 图谱深度增加，召回更多实体关联信息
- 召回从 72%-80% 提升至 **90%-95%**

### 阶段 3：降低响应时间（预期：14.8s → 3s-5s）

| 措施 | 实施方式 | 预期耗时下降 |
|------|----------|--------------|
| **LLM 流式首 token 优化** | 使用 `stream=true` + 缓存 system prompt，减少首 token 等待 | -3s ~ -5s |
| **检索并行化** | 向量/BM25/图谱三路并发（`asyncio.gather`） | -2s ~ -3s |
| **Embedding 缓存** | Redis/Memory 缓存 query embedding，30s TTL | -1s ~ -2s |
| **rerank 剪枝** | 阶段 1 已将输入从 16→8，rerank 计算量减少 50% | -1s ~ -2s |
| **换轻量 LLM（可选）** | 评估阶段用 `deepseek-ai/DeepSeek-V2.5` 替代 V3 | -2s ~ -4s |

**预期效果**：
- 综合优化后，总耗时从 14.8s 降至 **3s-5s**
- 如果仍需严格 ≤3s，可考虑评估阶段使用轻量模型

## 五、实施计划与验证步骤

### Step 1：精度优化（预计 15 分钟）

1. 修改 `app/core/config.py`：
   ```python
   retrieval_top_k: int = 8
   hybrid_vector_weight: float = 0.75
   hybrid_keyword_weight: float = 0.25
   rerank_top_k: int = 3
   ```
2. 重启后端服务
3. 执行 `python scripts/eval_comprehensive.py`
4. 检查精度是否 ≥ 60%，记录数据

### Step 2：召回优化（预计 20 分钟）

1. 修改 chunk 切分逻辑，增加 overlap=100
2. 修改 `graph_service.py`，增大 `max_depth=2`
3. 重新跑批，检查召回是否 ≥ 85%

### Step 3：速度优化（预计 30 分钟）

1. 检查三路检索是否已并行化，如未并行则改为 `asyncio.gather`
2. 增加 embedding 内存缓存（`functools.lru_cache` 或 Redis）
3. 评估阶段尝试 `deepseek-ai/DeepSeek-V2.5`
4. 重新跑批，检查耗时是否 ≤ 5s

### Step 4：最终验收跑批

1. 整合所有优化
2. 完整执行 `python scripts/eval_comprehensive.py`
3. 生成最终报告 `eval_comprehensive_report_optimized.md`
4. 对比基线 vs 优化后的逐题变化

## 六、风险与应对

| 风险 | 可能性 | 应对策略 |
|------|--------|----------|
| 精度提升但召回下降 | 中 | 阶段 1 和 2 同步进行，平衡 top_k 和 rerank_top_k |
| LLM 评分波动（LLM-as-Judge 不稳定） | 高 | 同一配置跑 2-3 次取平均 |
| 轻量 LLM 导致回答质量下降 | 低 | 先用 V3 验证优化方向，最后再换模型提速 |
| 修改 chunk 重叠需重新入库 | 中 | 阶段 2 如需重新切分，需预留 2 小时重新向量化 |

---

## 七、结论

当前系统距离工单验收标准存在**三个明确可优化的缺口**：

1. **精度 42% → 80%**：通过减少候选集、提高语义权重、收紧精排，**预计可达标**
2. **召回 72% → 90%**：通过 chunk 重叠、图谱深度、同义词扩展，**预计可达标**
3. **耗时 14.8s → 3s**：通过并行化、缓存、模型降级，**预计可接近或达标**

**建议优先执行阶段 1（精度优化）**，因为这是最大的短板，且改动最小（仅需改配置），30 分钟内可见效果。
